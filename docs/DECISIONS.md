## ADR-001. 심층 방어 — RLS deny-all + service_role 단일경로 + withAuth

- **맥락**: 실명·직장·학번·연락처를 다루는 동문 DB. 한 번의 실수로 전체 명부가 샐 수 있다. 운영자는 비개발 학생이라 "실수해도 안전한" 구조가 필요.
- **대안**: (A) RLS로 행 단위 정책을 정교하게 짠다 — 비개발자가 유지보수 불가, 정책 버그=유출. (B) RLS 끄고 서버 코드에서만 권한 검사 — 핸들러 하나라도 빠뜨리면 anon 키로 직통.
- **결정**: 세 겹. ① 전 테이블 RLS **ENABLE + 정책 0개(deny-all)** ② 모든 DB 접근은 `lib/supabase/admin.ts`(service_role) **단일 경로** ③ 각 핸들러를 `withAuth(role)`로 감싼다. ESLint가 컴포넌트의 admin import를 차단.
- **근거**: 정교한 RLS 정책은 비개발자가 못 건드린다. deny-all은 "켜고 잊는" 안전망 — 셋 중 하나가 뚫려도 나머지가 막는다. 권한 로직은 읽기 쉬운 TS(`withAuth`)에 모은다.
- **결과**: anon 키 직접 조회 시 0행(✅ 검증). 트레이드오프 — service_role이 만능키라 그 한 파일이 **단일 실패점**. 그래서 `server-only`+ESLint로 import 자체를 차단해 좁혔다.

## ADR-002. 단계 출시 (Phase 1 Core → 1.5 → 2 → 3)

- **맥락**: 동문 행사 데드라인 vs 6도메인(가입·디렉토리·연락·갤러리·구인·콘텐츠) 전부.
- **대안**: 6주에 6도메인 일괄 빅뱅 출시.
- **결정**: Core(가입·디렉토리·연락·안전·관리자)를 먼저 완성·검증하고 갤러리(1.5)→구인구직(2)→콘텐츠(3) 순으로 쌓는다. 단, **DB 스키마는 0001에 18테이블 전부 선설계**.
- **근거**: 핵심 가치(동문을 찾고 연결)부터 단단히. 스키마를 미리 깔아두면 추가 마이그레이션 없이 기능만 얹는다(실제 마이그레이션 1개로 P2·P3까지 구현).
- **결과**: 단계마다 typecheck/lint/build 그린 유지. 트레이드오프 — 선설계 테이블이 한동안 미사용 상태였으나, 결국 전부 채워짐.

## ADR-003. 개방형 회원제 (가입 = 회원, 사후 모더레이션)

- **맥락**: 초기 기획은 "검증된 동문만" — 추천/시드/관리자 승인 게이트.
- **대안**: 가입 신청 → 관리자 검증 → 승인 후 활동(폐쇄형).
- **결정**: 구글 로그인 + 최소정보 + 동의 3종이면 **즉시 `active` 회원**. 안전은 신고·차단·관리자 숨김으로 **사후** 처리.
- **근거**: 단톡방·교수 추천으로 이미 느슨히 연결된 모집단. 검증 큐는 비개발 운영자에게 과부하고 가입 마찰로 초기 활성도를 죽인다. 행사장에서 바로 가입시키는 시나리오에 맞다.
- **결과**: 온보딩 30초. 검증 게이트 코드는 제거됨([요구사항 §범위 외](spec/01-requirements.md#범위-외-의도적으로-컷)). 트레이드오프 — 비동문 유입 가능성을 모더레이션으로 감수.

## ADR-004. 커피챗 단순화 (오픈카톡 직접 + 이메일 제안)

- **맥락**: "커피챗 요청 → 수락/거절 → 일정 조율" 풀 플로우를 그릴 수 있었다.
- **대안**: 요청/수락/거절 상태머신 + 알림 왕복.
- **결정**: ① 본인이 공개한 **오픈카톡 링크로 직접 연결** ② 없으면 **이메일 제안을 서버 중계**(실 이메일 비노출, 1일 5건). `coffeechat_status`는 _가능성 신호_(open/monthly/offer_only/busy/private)일 뿐 워크플로 아님.
- **근거**: 요청/수락 플로우는 구현·운영 비용이 크고 "읽씹"으로 죽는다. 한국 사용자에겐 오픈카톡이 가장 자연스러운 1:1 채널. 자체 채팅을 만들 이유가 없다.
- **결과**: 연락 기능이 라우트 2개로 끝남. 트레이드오프 — 앱 내 대화 이력은 없음(의도적, [범위 외](spec/01-requirements.md#범위-외-의도적으로-컷)).

## ADR-005. PIPA를 "기능"으로 다룬다

- **맥락**: 실명·학번·직장 = 민감 개인정보. 한국 개인정보보호법(PIPA) 적용 대상.
- **대안**: 약관 한 장 붙이고 나중에 대응.
- **결정**: 분리 동의 3종 수집·기록(`consents`), 항목별 공개 토글(`field_visibility`), 탈퇴 시 식별정보 익명화/파기(`anonymized_at`), 타인 이메일·학번·오픈카톡 마스킹, 처리방침/약관 페이지를 **1급 기능**으로 구현.
- **근거**: 사후 추가가 가장 비싼 영역(데이터 모델·UX 다 바뀜). 신뢰가 곧 가입률인 실명 커뮤니티에선 프라이버시가 마케팅이기도 하다.
- **결과**: `consents`·soft delete·visibility가 스키마에 내장. 트레이드오프 — 처리방침 `【 】` 자리는 **법무 검토 + 보호책임자 지정** 후에야 실 출시 가능([잔여 R3·R4](spec/03-tasks.md#잔여-출시-전--코드-아님-운영-작업)).

## ADR-006. 무료 인프라 + 벤더 비종속

- **맥락**: 학생 동아리 예산 = 0. 운영 주체가 매 학기 바뀜.
- **대안**: 관리형 올인원 PaaS(유료), 또는 자체 서버.
- **결정**: Vercel Hobby + Supabase 무료 + Cloudflare R2(10GB) + Resend 무료. Supabase pause는 `/api/health` 핑으로 방지. 이미지·메일·DB를 표준 인터페이스 뒤에 둬 **이식 가능**하게.
- **근거**: 비용 0이 지속가능성. 한 벤더에 락인되면 가격·정책 변경 시 학생팀이 마이그레이션 불가.
- **결과**: 월 $0 운영. 트레이드오프 — 무료 한도·pause·콜드스타트 관리 필요(README 런북에 명시). R2 직접 PUT으로 Vercel 대역폭도 아낌.

## ADR-007. 갤러리는 운영자 큐레이션만 (자유 업로드 금지)

- **맥락**: 행사 사진 공유 요구 vs 초상권·부적절 이미지 리스크.
- **대안**: 회원 누구나 앨범/사진 업로드.
- **결정**: **운영자만** 앨범 생성·업로드. 공개하려면 `consent_confirmed`(게시 동의) 필수. 회원은 열람·라이트박스만.
- **근거**: 자유 업로드는 초상권 분쟁·모더레이션 부담·저장비를 부른다. 행사 기록은 본질적으로 큐레이션 콘텐츠. 비개발 운영자가 통제 가능한 범위로 묶는다.
- **결과**: 갤러리가 "공식 행사 아카이브" 성격으로 명확. 트레이드오프 — 회원 참여형 공유는 포기(의도적).

## ADR-008. 에디토리얼 다크 랜딩 + 코드 기반 비주얼

- **맥락**: "단톡방 대체" 수준이면 행사장에서 초라하다 — 휘어잡는 첫인상이 필요. 동시에 로그아웃이 숨겨져 있는 등 발견성 문제도 있었다.
- **대안**: (A) 흔한 라이트 "기업 홍보" 랜딩 — 반려(평범). (B) AI로 사람 얼굴 이미지 도배 — 가짜 인물의 신뢰 리스크.
- **결정**: 자체 **다크 에디토리얼 테마**(흑백 + 코발트 `#2D5BFF`, Instrument Serif 디스플레이), **코드 기반 추상 비주얼** + 선택적 AI 행사 커버(얼굴 없는 분위기 컷), 실데이터 카운트 노출.
- **근거**: 코드 기반 비주얼은 의존성·저작권·로딩 비용이 없고 톤을 정확히 통제. 가짜 얼굴 회피로 실명 커뮤니티의 진정성 유지([ADR-011](#adr-011-아바타커버--코드-그라데이션--얼굴-없는-ai-커버) 참조).
- **결과**: 행사 공개용으로 충분한 첫인상. 발견성 수정(로그아웃을 헤더 계정 메뉴로)도 함께. 트레이드오프 — 다크 랜딩 ↔ 라이트 앱 본문의 테마 이중성(의도적 분리).

## ADR-009. 마크다운 블로그 에디터 + 클라이언트 이미지 압축

- **맥락**: 초기 콘텐츠 작성이 "허접한 모달"이었다. 인터뷰·소식을 블로그 수준으로 쓰고 싶다.
- **대안**: 리치텍스트(WYSIWYG) 에디터 라이브러리 도입.
- **결정**: **마크다운** 에디터(툴바·본문 이미지 첨부·미리보기, react-markdown+remark-gfm) — 생성·수정 **동일 풀 에디터**. 이미지는 업로드 전 **클라이언트에서 canvas→WebP 압축**(scope별 maxDim) 후 R2 presigned PUT.
- **근거**: 마크다운은 저장 포맷이 투명(이식·백업 쉬움)하고 WYSIWYG 라이브러리 의존성·버그를 피한다. 클라이언트 압축으로 서버 대역폭 0, R2 용량·로딩 절약.
- **결과**: 블로그급 작성성. 트레이드오프 — 비개발 운영자가 마크다운 문법 학습 필요(툴바로 완화).

## ADR-010. 전수 감사 기반 수정

- **맥락**: 기능을 빠르게 쌓다 보니 "생성은 모달인데 수정은 풀 에디터" 같은 **경로 불일치**가 의심됐다.
- **대안**: 손에 잡히는 버그만 그때그때 수정.
- **결정**: 8개 병렬 에이전트로 **전 기능 감사** → 41건 도출 → 검증 후 High 4건 + 주요 Med/Low 수정(내 공고 화면, 차단 해제 UI, 인앱 알림 실제 생성, 알림 payload 통일, 검색어 인젝션 방어).
- **근거**: 사용자 한 명이 우연히 본 불일치는 빙산의 일각. 체계적 전수조사가 비개발 운영자에게 넘기기 전 품질을 보장.
- **결과**: 사용자 흐름의 구멍(예: 올린 공고를 다시 못 보는 문제)이 닫힘. 상세는 [태스크 T7](spec/03-tasks.md#t7-전수-감사-기반-수정).

## ADR-011. 아바타·커버 = 코드 그라데이션 + 얼굴 없는 AI 커버

- **맥락**: 프로필·콘텐츠에 시각적 채움이 필요하나 실제 인물 사진은 대부분 없다.
- **대안**: AI로 가짜 얼굴 생성해 채운다.
- **결정**: 사람 자리는 **코드 기반 결정론적 그라데이션 아바타**(이름→색), 콘텐츠 커버는 **얼굴 없는 분위기 AI 컷**(행사장·네트워킹·전시장·추상)만.
- **근거**: 실명 동문 플랫폼에서 **가짜 얼굴은 신뢰를 깬다**. 그라데이션은 결정론적(동일 사용자=동일 색)·무비용·무저작권.
- **결과**: 빈 프로필도 단정한 시각. 트레이드오프 — AI 커버 4종은 더미이므로 **출시 전 삭제 대상**([잔여 R2](spec/03-tasks.md#잔여-출시-전--코드-아님-운영-작업)).

## ADR-012. "인수인계 단순성"을 최종 판정 기준으로(메타)

- **맥락**: 매 학기 운영진이 **비개발 학생팀**으로 교체된다. 화려하지만 못 물려주는 코드는 죽은 코드.
- **대안**: 최신 추상화·라이브러리로 개발 속도/세련도 우선.
- **결정**: 모든 설계·기술 선택의 **최종 판정 기준 = "다음 사람이 이어받을 수 있는가"**. 무의존성 우선(자체 모션/라이트박스/아바타), 문자열 중앙화(`lib/messages`·`lib/labels`), 한 패턴 복제, 그리고 이 문서 세트(README+SDD+ADR).
- **근거**: 지속가능성 > 일시적 세련됨. 위 ADR들(deny-all·무료 인프라·커피챗 단순화·마크다운·코드 비주얼)이 모두 이 기준의 파생이다.
- **결과**: 학습 비용이 낮은 코드베이스 + 인계 문서. 트레이드오프 — 일부 "더 멋진" 추상화를 의도적으로 포기. 이것이 이 레포의 **북극성**이다.
