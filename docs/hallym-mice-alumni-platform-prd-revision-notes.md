## 2. 가장 중요한 변경 5가지

### 변경 1 — 보안 모델: "RLS 끄기"에서 "RLS 안전망 + 서버 검증" 심층 방어로 전환

- **무엇:** v1의 "RLS는 운영에서 제외하고 서버에서만 권한을 본다"를 폐기. v2는 "모든 테이블 RLS 켬(정책 없음 = 누구도 0행 = 전체 차단) + 데이터 접근은 서버 service_role 한 경로로만 + 모든 핸들러 첫 줄에서 `withAuth`로 권한 재검증"을 유일한 기본값으로 확정. connection string 직접 사용/PostgREST 권한 조작 옵션은 삭제.
- **왜:** Supabase는 테이블을 만들면 브라우저에 항상 노출되는 anon 키에 자동 권한을 준다. RLS를 끄면 그 키만으로 `/rest/v1/profiles`를 호출해 동문 전체의 실명·직장·학번이 그대로 읽힌다. v1은 "RLS 끄기"와 "브라우저 직접 접근 막기"를 혼동했다. "핸들러 권한 검사를 한 번도 안 빠뜨린다"는 가정은 시니어 팀도 실패하며, 매 학기 교체 + AI 코드 생성 환경에선 단일 실패점이 수십 개다.
- **기대효과:** 학생팀이 핸들러에서 권한 검사를 빠뜨려도 DB가 막아 PII가 새지 않는다. 게다가 정교한 역할별 정책을 짜는 게 아니라 "테이블마다 전체 차단 1줄"이라 **운영 난이도가 v1보다 오히려 낮다.**

### 변경 2 — 출시 범위: 6주 6도메인 일괄에서 Core 우선 단계 출시로

- **무엇:** 주차별 6도메인 일괄 빌드를 Phase 구조로 교체. Phase 1(인증/동의/검증 + 프로필 디렉토리·검색 + 커피챗 + 관리자 최소셋)을 먼저 완결, 구인구직은 Phase 2, 콘텐츠는 Phase 3. 갤러리·자동 통계 대시보드·뉴스레터 아카이브는 컷하고 외부 도구로 대체. Phase 진입은 "기능 완성"이 아니라 "실제 사용량 신호"로 판단(단, 차단선이 아니라 재평가 트리거).
- **왜:** v1은 "데이터·실연결이 1순위 생사 요인"이라 선언하면서 정작 데이터를 6주차에 몰아넣고 6도메인 완성을 1순위로 뒀다(자기모순). 비개발팀이 6주에 6도메인 빌드 + 실명 DB 운영 + 수기 영업을 동시 달성하는 것은 불가능하다.
- **기대효과:** 불가능한 범위가 Core 3~4주로 줄어 실제로 완주 가능해진다. 핵심 가치(검증된 디렉토리→커피챗 연결)를 끝까지 닫은 뒤, 나머지는 실제 사용 신호를 보고 붙이므로 콜드스타트 실패 시 개발 낭비를 막는다.

### 변경 3 — 동문 인증: 메모+직감에서 계층형 자동 검증으로

- **무엇:** "동문 확인용 메모 1줄 + 관리자 직감 승인"을 폐기. ①사전 등록 명부(학번+이름) 대조 또는 ②기존 승인 동문의 추천으로 가입하면 즉시 자동 승인, 둘 다 불가한 사람만 수동 심사 큐로 보낸다. 추천 남용 방지(1인 월 5건 상한·연대책임), 수동 승인자 관찰 기간(14일 연락처 제한), 명부 미확보 시 폴백까지 명시.
- **왜:** 가짜 1명이 들어오는 순간 "검증된 디렉토리"라는 제품의 존재 이유가 무너진다. 매 학기 바뀌는 운영자는 동문 얼굴을 모르므로 직감 승인은 기수가 갈수록 위험해진다. 또 운영 공백 시 승인 큐가 정체되면 대기자가 이탈한다.
- **기대효과:** 정상 동문은 승인 대기 없이 즉시 가입(전환율 방어), 위험 인물은 검증으로 차단. 운영자가 직감에 의존하지 않아도 검증이 구조적으로 작동한다.

### 변경 4 — PIPA 준수: 문서 작성 요구에서 동작하는 제품 기능으로

- **무엇:** "이용약관·처리방침 등 문서 4종을 만들라"는 추상 요구를, ①가입 화면의 분리 동의 체크박스 3개(이용약관 / 개인정보 수집·이용 / 프로필 공개) ②동의 시점·버전을 남기는 `consents` 테이블 ③탈퇴·동의철회 기능("프로필 비공개" vs "탈퇴/파기" 분리) ④처리방침 빈칸-채우기 템플릿(부록 C)으로 구체화.
- **왜:** 개인정보 보호법은 동의 입증·열람·삭제·파기를 강제한다. footer의 처리방침 링크만으로는 "동의받았다"는 입증이 불가능하고, v1에는 탈퇴/파기 기능 자체가 페이지 구조에도 티켓에도 없었다 — 명백한 법적 리스크.
- **기대효과:** 동의를 시점·버전과 함께 기록해 입증 가능해지고, 정보주체의 삭제·탈퇴 요구에 응할 수 있게 된다. 실명·직장 정보를 다루는 서비스로서 최소한의 법적 안전선을 확보한다.

### 변경 5 — 운영·인수인계 가능성 보강(이메일·비용·소유권·복구)

- **무엇:** ①Resend 트랜잭션 메일 + 앱 내 상태 배너로 승인/거절 알림 경로 확보 ②Vercel Pro·Supabase Pro를 출시 권장 전제로(0원은 명시적 리스크 수용 서명), 연간 운영비 표로 학과 예산 근거 제공 ③환경 분리(Prod/Preview)·주1회 백업 + 출시 전 복구 리허설 ④모든 외부 서비스를 학과/동문회 공용 계정 소유로 + 학기 교체 시 시크릿 재발급, 소유권 매트릭스·3대 장애 런북·security.md 5계명 추가.
- **왜:** v1은 승인 후 알림 수단이 없어 "승인해도 회원은 모름"이 기본값이었고, 비용/약관 리스크(Hobby 상업 차단, 무료 자동 정지)를 "사용량 확인" 한 줄로 넘겼으며, 키가 졸업생 개인 계정에 묶이면 연락 두절 시 서비스가 인질이 되고, 백업은 "확인"만 있어 복구가 검증되지 않았다.
- **기대효과:** 회원이 승인 사실을 알게 되고, 비용·약관 사고를 사전에 막으며, 매 학기 운영진이 바뀌어도 시크릿 로테이션·소유권 매트릭스·런북으로 인수인계가 구조적으로 작동한다.

## 3. 그대로 유지한 강점

원본이 잘한 부분은 후퇴시키지 않고 그대로 살렸다.

- **외부 도구 위임 원칙:** 채팅=오픈카톡, 영상=YouTube, 뉴스레터=Stibee/Maily, 결제=외부 링크. 유지보수 부담과 보안 공격면을 동시에 줄이는 핵심 전략. "후퇴가 아니라 전략"이라는 결론 유지.
- **프로필 중심 정보구조:** 게시판 모음이 아니라 검증된 동문 디렉토리. 모든 부가 기능을 프로필에 연결.
- **브라우저 DB 직접 접근 금지 + 민감 데이터 서버 집중:** 방향은 옳다. RLS "대신"을 "함께"로 바꿔 그대로 살림.
- **service_role 키 서버 전용 격리, `NEXT_PUBLIC_`엔 공개값만.**
- **관리자 화면으로 운영 캡슐화 + admin_logs 감사 추적:** 운영자가 Supabase 콘솔을 직접 만지지 않게 한다.
- **삭제보다 숨김(soft delete) + 상태값 표준화:** pending/approved/rejected/published/hidden/closed.
- **개인정보 최소 수집:** 주민번호·생년월일·전화번호·주소·민감정보 미수집.
- **의존성 기반 티켓 백로그 + 티켓당 Preview 모바일 확인 루프:** 바이브코딩에 적합한 작업 분해.
- **프롬프트 템플릿([목표/사용자/데이터/권한/UI/완료기준/주의사항]):** 여기에 정확한 컬럼명·제약·상태만 보강.
- **학기별 인수인계 체크리스트 골격:** 핵심 제약을 루틴화(시크릿 로테이션·소유권 매트릭스로 보강).

## 4. 남은 의사결정 / 사용자 확인 필요 사항

아래는 PM(기획 의사결정자)이 직접 확인·결정해야 진행이 막히지 않는 항목이다.

1. ✅ **[결정됨 — 예산 0원]** 무료 티어로 출시한다. PRD §9.2를 "무료가 기본 계획"으로 개정했고 무료의 두 지뢰를 설계로 해결: ①Vercel은 호스트 종속 금지 + Cloudflare/Netlify 이전 경로 문서화(Phase 1은 결제·공고 없는 비영리라 Hobby 사용이 합리적, 공고가 생기는 Phase 2 진입이 호스트 재평가 트리거), ②Supabase는 pause 방지 핑(cron-job.org `/api/health`, 신규 티켓 T-306)을 Phase 1 기본 설정으로. 유료 전환은 예산 확보·한도 초과 시의 업그레이드일 뿐 출시 조건 아님.
2. ✅ **[결정 — 개방형 회원제로 전환]** 가입=회원=즉시 풀 사용. 검증 게이트·추천·시드(30명)·수동 심사를 **전부 제거**(v2.3). 안전은 사후 모더레이션(신고/정지/차단). 학교 명부는 나중에 선택적 **'인증 배지'(`is_verified`, 비차단 정보용)**로만(§6.1b). → 시드 담당·검증 설계 결정은 더 이상 불필요.
3. **개인정보/법무 검토:** 처리방침·이용약관(부록 C 템플릿 기반)을 가능하면 법무 검토. 개인정보처리자(동문회/학과 공식 단위)와 개인정보 보호책임자 1명을 지정해야 함(§8.3, §14.1).
4. ✅ **[결정 — 프로젝트 공용 계정 1개]** 모든 외부 서비스(GitHub/Supabase/Vercel/Cloudflare R2/Resend/도메인/Google OAuth)를 졸업해도 넘길 수 있는 **전용 공용 계정 1개**로 기술 담당 1명이 셋업(개인 계정 금지, MFA, 학기말 시크릿 로테이션, 부록 E). → 남은 것: 그 공용 계정 주소를 실제로 만들기.
5. ⬜ **출시 게이트 수치:** Phase 2 착수 신호(예: 활성 회원 50명 + 프로필 완성 30명 + 커피챗 클릭 누적 40건)와 성공 지표 목표치(§2.2)를 운영진과 합의.
6. **트랜잭션 메일 발신 도메인:** Resend 도메인 인증(SPF/DKIM/DMARC)에 쓸 도메인 또는 서브도메인(mail.도메인) 확보(§9.1, H1).
7. **이미지 서빙 도메인(R2):** Cloudflare R2 공개 읽기를 `r2.dev` 기본 도메인으로 갈지 커스텀 도메인(예: `img.도메인`, Cloudflare CDN)으로 갈지. 버킷 CORS 허용 origin = 배포 도메인(§9.2).
8. **행사 사진 게시 동의 담당:** 갤러리 앨범 공개 전 피사체 동문 게시 동의 확보·삭제 요청 처리를 누가(콘텐츠 담당) 책임질지(§6.5, §8.3).

## 6. v2.1~v2.4 PM 추가 결정 (이 노트 §1~§3 작성 후 누적 반영 — 현재 정본 = v2.4)

PM과 반복 협의하며 PRD를 단순화·구체화했다. 매 변경마다 전수 적대적 검수로 구모델 잔재 0건을 확인했다. 큰 줄기는 **"게이트를 걷어내고 각자 자기 데이터를 직접 통제, 안전은 사후 모더레이션"**이다.

- **v2.1 (예산·명부):** 예산 0원 → 무료 티어 + 무료의 두 지뢰(Vercel 약관·Supabase pause)를 설계로 해결. 명부 미확보 확정 반영.
- **v2.2 (스토리지·갤러리):** 이미지 스토리지 = Cloudflare R2(무료 10GB·egress 0, `lib/storage` 어댑터) 처음부터. 갤러리 Phase 1.5 복원(운영자 큐레이션, 행사 사진 게시 동의/PIPA).
- **v2.3 (개방형 회원제 — v2.2의 "점진적 신뢰"를 대체):** "가입=회원=즉시 풀 사용". 검증 게이트·추천 시스템·시드 30명·수동 심사 큐·미인증/인증 2단계·teaser를 **전부 제거**. `status`=`active/suspended/withdrawn`, `trust_level` 제거 → `is_verified`(선택적·비차단 배지). 안전은 신고/정지/차단(사후). `referrals`·`verification_requests` 테이블 삭제, `verification_roster`는 미래 배지 옵션으로만.
- **v2.4 (오픈카톡 직접 + 커피챗 단순화):** 오픈카톡 URL은 본인이 등록·공개 여부를 정하므로 공개 시 회원에게 **직접 표시**(수락 게이트 없음). 구조화된 커피챗 요청/수락/거절 플로우와 `coffeechat_requests` 테이블 제거 → 직접 링크 모델. 하단 탭 4→3(홈/동문/내정보). **제안 이메일만 서버 중계**(실제 개인 이메일 보호). 차단(block)·제안 폴백 유지.

> **순효과:** 만들 것이 크게 줄었다(추천 트리·심사 큐·요청/수락 플로우·2단계 게이팅·teaser 제거). 비개발팀이 완주·운영·인수인계하기 훨씬 쉬워졌고, 핵심 가치(동문 찾기 → 오픈카톡 직접 연결)는 그대로다. 보안(RLS deny-all·withAuth·service_role)·PIPA(동의·탈퇴·초상권)·무료 인프라(R2·pause 핑)는 끝까지 유지.
