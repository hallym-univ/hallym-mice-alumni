# 한림대 MICE 동문 플랫폼 구현 기획서

문서 버전: v1.0  
작성일: 2026-06-08  
작성 관점: 시니어 PM / 비개발 학생팀의 AI 구현 및 운영 유지보수 기준

## 1. 결론

이 프로젝트는 기능별 실험 서비스가 아니라, 처음부터 하나의 통합 플랫폼으로 만든다.

다만 "전부 직접 개발"은 하지 않는다. 동문 데이터와 권한이 필요한 핵심 기능은 직접 만들고, 채팅, 뉴스레터 발송, 영상 호스팅, 결제처럼 유지보수 비용이 큰 기능은 외부 도구로 연결한다. 이것이 비개발 학생팀이 AI로 구현하고 다음 기수에게 인수인계할 수 있는 가장 현실적인 방식이다.

제품의 중심은 커뮤니티 게시판이 아니라 동문 프로필이다.

> 한림대 MICE 동문 플랫폼은 검증된 동문을 찾고, 커피챗, 제안, 채용, 콘텐츠, 행사 기록으로 연결되는 신뢰 기반 동문 네트워크 운영 시스템이다.

이 문서는 기존 회의록과 Google Docs의 내용을 참고했지만, "독립 MVP를 여러 개 만들고 나중에 합친다"는 방식은 채택하지 않는다. 대신 단일 코드베이스, 단일 데이터베이스, 단일 관리자 시스템으로 처음부터 통합 설계한다.

## 2. 제품 목표

### 2.1 비즈니스 목표

1. 한림대 MICE 계열 재학생, 졸업생, 동문 기업을 하나의 신뢰 네트워크로 묶는다.
2. 재학생에게는 진로, 취업, 실무 조언을 얻을 수 있는 연결 통로를 제공한다.
3. 현직 동문에게는 업계 네트워크, 사업 제안, 채용 후보자 탐색, 후배 지원의 접점을 제공한다.
4. 기업 대표 및 채용 담당자에게는 검증된 인재와 동문 기반 협업 기회를 제공한다.
5. 학교와 동문회 운영진에게는 동문 DB, 콘텐츠, 채용 정보, 행사 기록을 유지할 수 있는 운영 인프라를 제공한다.

### 2.2 제품 성공 기준

초기 3개월 기준:

| 지표 | 목표 |
| --- | --- |
| 승인 회원 수 | 100명 이상 |
| 핵심 프로필 완성률 | 승인 회원의 70% 이상 |
| 커피챗/오픈카톡 클릭 | 월 30건 이상 |
| 구인구직 외부 지원 링크 클릭 | 월 30건 이상 |
| 신규 공고 등록 | 월 10건 이상 |
| 인터뷰/콘텐츠 발행 | 월 2건 이상 |
| 관리자 주간 운영 시간 | 주 3시간 이하 |

### 2.3 제품이 풀어야 하는 핵심 문제

| 문제 | 제품적 해법 |
| --- | --- |
| 동문이 어디서 무엇을 하는지 모른다 | 검증된 프로필 디렉토리와 검색 |
| 모르는 선후배에게 연락하기 부담스럽다 | 커피챗 가능 상태와 오픈카톡 연결 |
| MICE 업계 채용이 추천과 상시 채용 중심이다 | 동문 기반 구인구직/기회 게시판 |
| 기존 카톡방 정보가 흘러가고 검색되지 않는다 | 플랫폼 내 구조화된 공고, 콘텐츠, 아카이브 |
| 운영 주체가 불명확하면 플랫폼이 방치된다 | 관리자 화면, 운영 로그, 주간 운영 루틴 |
| 비개발 학생들이 유지보수해야 한다 | 단일 스택, 단순 권한, 외부 도구 적극 활용 |

## 3. 제품 원칙

### 3.1 반드시 지킬 원칙

1. 모바일 우선으로 만든다.
2. 프로필을 모든 기능의 중심 데이터로 둔다.
3. 내부 채팅은 만들지 않고 오픈카톡 링크로 연결한다.
4. 영상 직접 업로드는 만들지 않고 YouTube 링크 임베드로 처리한다.
5. 뉴스레터 발송 시스템은 만들지 않고 Stibee, Maily, Google Form 등 외부 도구로 연결한다.
6. 결제 시스템은 만들지 않고 계좌 안내 또는 외부 결제 링크로 처리한다.
7. 학번, 직장, 연락 가능 여부 등 개인정보가 포함되므로 RLS 대신 서버 API에서 권한을 통제한다.
8. Supabase 콘솔을 운영자가 직접 만지지 않도록 관리자 화면을 만든다.
9. AI가 코드를 생성하더라도 모든 기능은 명확한 완료 기준을 통과해야 한다.
10. 다음 기수가 이어받을 수 있도록 README, 운영 매뉴얼, 환경변수 목록, DB 스키마를 남긴다.

### 3.2 만들지 않을 것

| 제외 기능 | 제외 이유 | 대체 방식 |
| --- | --- | --- |
| 자체 1:1 채팅 | 알림, 신고, 보관, 개인정보 이슈로 유지보수 부담이 큼 | 오픈카톡 링크 |
| 자체 푸시 알림 | 모바일 앱/토큰/권한 관리가 필요함 | 이메일, 카톡방 공지, 외부 링크 |
| 네이티브 앱 | 앱스토어 심사, 버전 업데이트, OS 대응 부담 | 모바일 웹 + PWA |
| 영상 직접 업로드 | 스토리지/트랜스코딩 비용과 운영 부담 | YouTube 임베드 |
| 추천 알고리즘 | 데이터가 적은 초기에는 효과보다 복잡도가 큼 | 검색/필터/운영자 큐레이션 |
| 전자결제 | 법무/정산/환불/영수증/보안 부담 | 계좌 안내 또는 외부 결제 링크 |
| LMS/과제 제출 | 동문 플랫폼의 핵심 행동과 거리가 있음 | 필요 시 외부 LMS 링크 |

## 4. 사용자 정의

### 4.1 재학생

목표:

- 선배의 직무 경험을 알고 싶다.
- 인턴, 취업, 프로젝트 기회를 찾고 싶다.
- 모르는 선배에게 부담 없이 연락하고 싶다.

필요 기능:

- 동문 검색
- 프로필 상세
- 커피챗 가능 상태
- 오픈카톡 연결
- 구인구직 게시판
- 인터뷰 콘텐츠

### 4.2 일반 현직 동문

목표:

- 자신의 소속과 전문성을 알리고 싶다.
- 업계 동문과 연결되고 싶다.
- 후배에게 적당한 범위에서 도움을 주고 싶다.
- 이직, 협업, 프로젝트 정보를 얻고 싶다.

필요 기능:

- 내 프로필 관리
- 공개 범위 설정
- 커피챗 가능 상태 설정
- 공고/기회 제보
- 동문 검색
- 콘텐츠/행사 기록 열람

### 4.3 기업 대표/채용 담당자/시니어 동문

목표:

- 검증된 후배와 인재를 찾고 싶다.
- 회사와 프로젝트를 홍보하고 싶다.
- 산학 협업 가능성을 탐색하고 싶다.

필요 기능:

- 동문/학생 검색
- 구인구직 공고 등록
- 회사/직무 태그 기반 필터
- 제안 보내기
- 기업 관련 인터뷰/콘텐츠 연결

### 4.4 학교/동문회 운영진

목표:

- 동문 DB를 유지하고 싶다.
- 인증된 사용자만 내부 정보를 보게 하고 싶다.
- 채용, 콘텐츠, 행사 기록을 관리하고 싶다.
- 운영자가 바뀌어도 플랫폼을 유지하고 싶다.

필요 기능:

- 회원 승인/거절
- 프로필 숨김/신고 처리
- 공고 관리
- 콘텐츠 관리
- 갤러리 관리
- 기본 통계
- 관리자 로그

## 6. 기능 요구사항

### 6.1 인증 및 회원 승인

목표:

동문/재학생/학교 관계자만 내부 정보를 볼 수 있게 한다.

요구사항:

1. Google OAuth 로그인만 제공한다.
2. 이메일/비밀번호 회원가입은 제공하지 않는다.
3. 최초 로그인 후 가입 정보 입력 화면으로 보낸다.
4. 사용자는 다음 정보를 입력한다.
   - 이름
   - 역할: 재학생, 졸업생, 교수/운영진, 외부 파트너
   - 학번 또는 입학/졸업연도
   - 학과/전공
   - 현재 소속
   - 직무/관심 분야
   - 동문 확인용 메모
5. 가입 직후 상태는 `pending`이다.
6. 관리자가 승인하면 `approved`가 된다.
7. `approved` 사용자만 프로필 상세, 오픈카톡 링크, 구인구직 상세를 볼 수 있다.
8. 거절된 사용자는 사유를 확인하고 재요청할 수 있다.

완료 기준:

- 비로그인 사용자는 내부 페이지 접근 시 로그인 화면으로 이동한다.
- 로그인했지만 승인 전인 사용자는 승인 대기 화면만 볼 수 있다.
- 승인된 사용자만 동문 상세와 공고 상세를 볼 수 있다.
- 관리자가 승인/거절하면 상태가 즉시 반영된다.

### 6.2 동문 프로필/디렉토리

목표:

사용자가 관심 회사, 직무, 분야, 졸업연도 기준으로 동문을 찾고 연락할 수 있게 한다.

프로필 필드:

| 필드 | 필수 | 공개 범위 |
| --- | --- | --- |
| 이름 | 필수 | 인증 회원 |
| 프로필 사진 | 선택 | 인증 회원 |
| 역할 | 필수 | 인증 회원 |
| 입학/졸업연도 | 선택 | 인증 회원 |
| 회사/기관 | 필수 | 인증 회원 |
| 직무 | 필수 | 인증 회원 |
| 분야 태그 | 필수 | 인증 회원 |
| 한 줄 소개 | 선택 | 인증 회원 |
| 경력 요약 | 선택 | 인증 회원 |
| 커피챗 상태 | 필수 | 인증 회원 |
| 오픈카톡 URL | 선택 | 인증 회원 |
| 이메일 제안 허용 여부 | 선택 | 인증 회원 |
| 공개 여부 | 필수 | 관리자/본인 |

커피챗 상태 옵션:

- 가능
- 월 1회 가능
- 채용/업무 제안만 가능
- 지금은 어려움
- 비공개

목록 화면:

- 검색창
- 회사/기관 필터
- 직무 필터
- 분야 태그 필터
- 졸업연도/기수 필터
- 커피챗 가능 필터
- 프로필 카드 리스트

프로필 카드 표시:

- 이름
- 회사/기관
- 직무
- 분야 태그
- 커피챗 상태
- 한 줄 소개

프로필 상세 액션:

- 오픈카톡으로 커피챗
- 이메일로 제안 보내기
- 프로필 링크 복사
- 신고하기

완료 기준:

- 사용자가 내 프로필을 생성/수정할 수 있다.
- 필수값이 없으면 저장되지 않는다.
- 인증 회원이 동문을 검색하고 필터링할 수 있다.
- 오픈카톡 URL이 없는 프로필에는 커피챗 버튼이 비활성화된다.
- 오픈카톡 클릭 이벤트가 기록된다.
- 본인 또는 관리자만 프로필을 수정할 수 있다.

- 마감일이 지난 공고는 목록에서 마감 상태로 보인다.

### 7.4 RLS 운영 제외형 권한 설계

운영 난이도를 낮추기 위해 v1에서는 Supabase RLS를 기능별 권한 정책으로 운영하지 않는다. 대신 브라우저가 Supabase 테이블을 직접 읽고 쓰지 못하게 하고, 모든 민감 데이터 접근을 Next.js 서버 API route 또는 Server Action으로 모은다.

중요한 구분:

- 하지 않는 것: 테이블별/역할별 RLS 정책을 계속 관리하는 방식
- 반드시 하는 것: 브라우저에서 DB 테이블 직접 접근을 막고, 서버에서 권한 검사 후 DB 작업 수행

RLS를 안 쓰는 경우에도 권한 자체가 사라지면 안 된다. 권한은 RLS가 아니라 서버 코드에서 통제한다.

이 방식의 원칙:

1. 브라우저 클라이언트에서는 Supabase Auth만 사용한다.
2. 브라우저에서 `profiles`, `jobs`, `articles`, `albums`, `reports`, `events`, `admin_logs`를 직접 query/insert/update/delete 하지 않는다.
3. DB 조회와 수정은 `/api/*` route 또는 Server Action에서만 처리한다.
4. 서버는 로그인 세션을 확인하고, 사용자의 `profiles.status`, `role`, 관리자 여부를 검사한 뒤 DB 작업을 수행한다.
5. Supabase `service_role` key는 서버 환경변수에만 두고 클라이언트 번들에 절대 포함하지 않는다.
6. 운영 데이터 테이블은 브라우저에 직접 노출되는 Supabase 자동 API 접근을 막는 방향으로 구성한다.
7. 구현 방식은 아래 두 가지 중 하나를 선택한다.
   - 권장안 A: Next.js 서버에서만 Postgres connection string 또는 Supabase server client를 사용하고, 브라우저에서는 Auth만 사용한다. 자동 API 접근은 grant/revoke 또는 schema 설정으로 차단한다.
   - 대안 B: 복잡한 RLS 정책은 만들지 않고, RLS를 "전체 차단 안전장치"로만 사용한다. 실제 권한 판단은 서버 API에서 한다.
8. 정식 운영 전 보안 점검 때 선택한 방식이 브라우저 직접 접근을 실제로 차단하는지 확인한다.

권한은 코드 레벨에서 아래 기준으로 검사한다.

| 테이블 | READ | CREATE | UPDATE | DELETE/HIDE |
| --- | --- | --- | --- | --- |
| `profiles` | 승인 회원은 공개 프로필 열람, 본인은 자신의 전체 정보 열람 | 로그인 사용자는 본인 프로필 생성 | 본인 또는 관리자 | 관리자 |
| `verification_requests` | 본인 또는 관리자 | 로그인 사용자 | 본인 재요청 또는 관리자 | 관리자 |
| `jobs` | 승인 회원은 published 열람 | 승인 회원 | 작성자 draft/pending 수정, 관리자는 전체 수정 | 관리자 |
| `articles` | 승인 회원은 published 열람 | 콘텐츠 운영자/관리자 | 작성자/관리자 | 관리자 |
| `albums` | 승인 회원은 공개 앨범 열람 | 운영자/관리자 | 운영자/관리자 | 관리자 |
| `events` | 관리자만 집계 열람 | 서버 API에서만 기록 | 불가 | 관리자 |
| `admin_logs` | 관리자만 | 서버에서만 | 불가 | 불가 |

주의:

- Supabase `service_role` key는 절대 브라우저 코드에 노출하지 않는다.
- `NEXT_PUBLIC_` 환경변수에는 공개되어도 되는 값만 넣는다.
- 관리자 작업은 서버 API route 또는 Server Action에서 처리한다.
- RLS 정책으로 기능 권한을 운영하지 않는 대신, API handler마다 `requireApprovedUser`, `requireAdmin`, `requireOwnerOrAdmin` 같은 공통 권한 헬퍼를 반드시 사용한다.
- 구현자가 "RLS를 안 쓰니까 Supabase 클라이언트에서 바로 조회해도 된다"고 판단하면 안 된다. 그 방식은 개인정보 노출 위험이 크다.

- Vercel 환경변수는 Production, Preview, Development 환경별로 나눠 관리할 수 있다.

## 11. 화면별 요구사항

### 11.1 랜딩

목적:

- 서비스가 무엇인지 설명하고 로그인으로 유도한다.

필수 요소:

- 서비스명
- 한 줄 설명
- 로그인 버튼
- 주요 가치 3개: 동문 찾기, 커피챗, 기회 탐색
- 개인정보 처리방침/이용약관 링크

### 11.2 홈

목적:

- 로그인 후 사용자가 가장 중요한 행동으로 바로 이동하게 한다.

필수 요소:

- 검색창
- 커피챗 가능한 추천 동문
- 최신 공고
- 최신 인터뷰
- 최근 행사 기록

### 11.3 동문 목록

목적:

- 사용자가 원하는 동문을 빠르게 찾는다.

필수 요소:

- 검색창
- 필터
- 카드 목록
- 빈 상태 안내
- 내 프로필 등록 CTA

### 11.4 프로필 상세

목적:

- 신뢰할 수 있는 정보를 보고 연락/제안으로 이어진다.

필수 요소:

- 기본 정보
- 경력/소개
- 태그
- 커피챗 상태
- 오픈카톡 버튼
- 제안 이메일 버튼
- 신고 버튼

### 11.5 공고 목록/상세

목적:

- 실질적인 채용/기회 정보를 찾고 지원 링크로 이동한다.

필수 요소:

- 유형 필터
- 마감 상태
- 북마크
- 상세 정보
- 외부 지원 버튼
- 작성자 프로필

### 11.6 콘텐츠 목록/상세

목적:

- 동문 이야기를 읽고 프로필 또는 뉴스레터 구독으로 이동한다.

필수 요소:

- 콘텐츠 카드
- 태그
- 관련 프로필
- 구독 링크

### 11.7 관리자 대시보드

목적:

- 운영자가 오늘 해야 할 일을 바로 파악한다.

필수 요소:

- 승인 대기 수
- 신고 대기 수

## 13. 바이브코딩 작업 방식

### 13.1 절대 금지 프롬프트

아래처럼 요청하지 않는다.

```text
한림대 동문 플랫폼 전체를 만들어줘.
```

이렇게 요청하면 AI가 권한, DB, 화면, 관리자, 보안을 뒤섞어서 불안정한 코드를 만들 가능성이 높다.

### 13.2 권장 프롬프트 구조

모든 요청은 아래 구조로 작성한다.

```text
[목표]
무엇을 만들지 한 문장으로 설명한다.

[사용자]
누가 이 기능을 쓰는지 설명한다.

[데이터]
사용할 테이블과 필드를 설명한다.

[권한]
누가 볼 수 있고, 누가 만들고, 누가 수정할 수 있는지 설명한다.

[UI]
모바일 화면에서 어떤 요소가 보여야 하는지 설명한다.

[완료 기준]
어떤 조건을 통과해야 끝났다고 볼지 설명한다.

[주의사항]
비밀키 노출 금지, 서버 권한 검사 유지, 클라이언트 DB 직접 접근 금지, 에러/빈 상태 처리 등을 적는다.
```

### 13.3 예시 프롬프트: 프로필 디렉토리

```text
[목표]
Next.js App Router와 Supabase를 사용해서 동문 프로필 디렉토리 화면을 구현해줘.

[사용자]
승인된 회원이 회사, 직무, 태그로 동문을 검색하고 프로필 상세로 이동한다.

[데이터]
profiles 테이블을 사용한다. name, organization, position, bio, coffeechat_status, tags, is_public, status 필드를 사용한다.

[권한]
approved 사용자만 목록과 상세를 볼 수 있다. pending 사용자는 승인 대기 화면으로 보내고, 비로그인 사용자는 로그인 화면으로 보낸다. 본인과 관리자만 프로필을 수정할 수 있다.

[UI]
모바일 우선 화면이다. 상단 검색창, 필터 칩, 프로필 카드 리스트, 빈 상태 안내, 내 프로필 수정 버튼을 만든다.

[완료 기준]
검색어로 이름/회사/직무가 필터링된다. 커피챗 가능한 사용자만 보는 필터가 있다. 프로필 카드를 누르면 상세로 이동한다. 빈 결과일 때 안내가 보인다.

[주의사항]
Supabase service role key를 클라이언트에서 쓰지 마. 브라우저에서 Supabase table query를 직접 하지 말고, Next.js 서버 API에서 세션과 approved 상태를 검사한 뒤 데이터를 내려줘. 에러 상태와 로딩 상태를 구현해줘.
```

### 13.4 예시 프롬프트: 관리자 승인

```text
[목표]
관리자가 가입 승인 요청을 확인하고 승인/거절할 수 있는 화면을 만들어줘.

[사용자]
관리자 이메일 목록에 포함된 사용자만 접근할 수 있다.

[데이터]
profiles와 verification_requests를 사용한다. pending 상태의 사용자를 목록으로 보여준다.

[권한]
관리자만 접근 가능하다. 일반 사용자가 /admin/users에 접근하면 홈으로 리다이렉트한다. 승인 작업은 서버 API에서 처리한다.

[UI]
모바일과 데스크톱 모두에서 쓸 수 있는 관리자 테이블을 만든다. 이름, 이메일, 역할, 학번/연도, 소속, 요청 메모, 승인/거절 버튼이 보여야 한다.

[완료 기준]
승인하면 profiles.status가 approved로 바뀐다. 거절하면 rejected로 바뀌고 사유가 저장된다. 모든 작업은 admin_logs에 기록된다.

[주의사항]
관리자 권한 체크를 클라이언트 UI 숨김에만 의존하지 마. 서버에서도 반드시 검사해줘.
```

## 15. QA 체크리스트

### 15.1 모바일 QA

- iPhone 375px 폭에서 텍스트가 잘리지 않는다.
- Android Chrome에서 버튼과 입력창이 정상 작동한다.
- 하단 내비게이션이 화면을 가리지 않는다.
- 검색/필터 사용 후 목록이 자연스럽게 갱신된다.
- 긴 회사명/직무명이 카드 레이아웃을 깨지 않는다.
- 외부 링크는 새 탭 또는 안전한 방식으로 열린다.

### 15.2 권한 QA

- 비로그인 사용자는 내부 페이지 접근 불가.
- pending 사용자는 승인 대기 화면만 접근 가능.
- approved 사용자는 내부 페이지 접근 가능.
- 일반 사용자는 관리자 화면 접근 불가.
- 본인은 자기 프로필만 수정 가능.
- 관리자는 승인/숨김/마감 처리 가능.
- 브라우저 코드에서 Supabase table query를 직접 호출하는 기능이 없어야 한다.

### 15.3 기능 QA

- Google 로그인 성공.
- 프로필 생성/수정 성공.
- 프로필 검색/필터 성공.
- 오픈카톡 클릭 이벤트 기록.
- 공고 등록/승인/마감 성공.
- 지원 링크 클릭 이벤트 기록.
- 인터뷰 글 발행/숨김 성공.
- 갤러리 이미지 업로드 성공.
- YouTube 임베드 성공.
- 신고 접수/처리 성공.

### 15.4 운영 QA

- 운영자가 관리자 화면만 보고 신규 회원을 승인할 수 있다.
- 운영자가 관리자 화면만 보고 공고를 마감할 수 있다.
- README를 보고 로컬 실행이 가능하다.
- 환경변수 목록이 문서화되어 있다.
- 알려진 버그 목록이 있다.
- 개인정보 처리방침 초안이 있다.

1. 로그인/승인/권한 흐름이 정상 작동한다.
2. 승인 회원만 내부 정보를 볼 수 있다.
3. 동문 프로필 30개 이상이 등록되어 있다.
4. 구인구직 공고 10개 이상이 등록되어 있다.
5. 인터뷰 콘텐츠 3개 이상이 등록되어 있다.
6. 관리자 화면에서 승인, 공고, 콘텐츠, 신고 처리가 가능하다.
7. 모바일 QA를 통과했다.
8. 핵심 이벤트 로깅이 작동한다.
9. 개인정보 처리방침과 이용약관 링크가 있다.
10. README와 운영 매뉴얼이 있다.
11. 환경변수가 GitHub에 노출되지 않았다.
12. 브라우저에서 DB 테이블에 직접 접근하지 않아도 서비스가 정상 작동한다.

## 21. AI 코드 리뷰 체크포인트

AI가 생성한 코드는 아래 항목을 통과해야 한다.

### 21.1 보안

- `service_role` key가 클라이언트 코드에 없다.
- `NEXT_PUBLIC_` 환경변수에 비밀값이 없다.
- 관리자 권한 체크가 클라이언트 숨김 처리에만 의존하지 않는다.
- 브라우저에서 Supabase table query를 직접 호출하지 않는다.
- 서버 API마다 세션, 승인 상태, 관리자 여부를 검사한다.
- 사용자가 다른 사람의 프로필을 수정할 수 없다.

### 21.2 UX

- 로딩 상태가 있다.
- 에러 상태가 있다.
- 빈 목록 상태가 있다.
- 모바일에서 버튼이 잘리지 않는다.
- 외부 링크 클릭 전 사용자가 무엇을 누르는지 알 수 있다.

### 21.3 운영

- 관리자가 Supabase 콘솔 없이 주요 작업을 처리할 수 있다.
- 상태값이 명확하다: pending, approved, rejected, published, hidden, closed.
- 삭제보다 숨김을 우선한다.
- 관리자 작업 기록이 남는다.
- README만 보고 새 운영자가 실행 흐름을 이해할 수 있다.

## 22. PM 최종 판단

이 플랫폼의 성패는 "기능을 많이 만들었는가"가 아니라 "동문 데이터가 신뢰 가능하고, 실제 연결과 기회가 발생하는가"에 달려 있다.

따라서 구현 우선순위는 다음과 같다.

1. 권한과 인증
2. 프로필 디렉토리
3. 커피챗/제안 연결
4. 구인구직/기회 게시판
5. 관리자 화면
6. 콘텐츠/갤러리
7. 통계/운영 자동화

전체 플랫폼을 한 번에 만들더라도, 제품의 중심은 끝까지 프로필이어야 한다. 구인구직, 인터뷰, 갤러리, 행사 기록은 모두 프로필과 연결되어야 한다. 그래야 게시판 모음이 아니라 동문 네트워크가 된다.

비개발 학생팀이 AI로 구현하는 경우 가장 중요한 것은 기술 욕심을 줄이는 것이 아니라 운영 리스크를 줄이는 것이다. 내부 채팅, 네이티브 앱, 직접 결제, 직접 영상 호스팅을 제외하는 것은 후퇴가 아니라 유지 가능한 제품을 만들기 위한 제품 전략이다.
